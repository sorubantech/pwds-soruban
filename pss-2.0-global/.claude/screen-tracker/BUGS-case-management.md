## CRITICAL

### [x] C-1 — Fund pool over-committable via check-then-act race (TOCTOU) — `FIXED (session 1)`
- **Where**: `ServiceLogFundingGuard.cs:36-42,71-77` (unguarded `SumAsync`); reachable via `CreateBeneficiaryServiceLog` + `UpdateBeneficiaryServiceLog.cs:89-93`.
- **Problem**: pool checked then written with no lock / no concurrency token → two concurrent logs both pass, both spend, pool goes negative. Zero concurrency tokens anywhere in CaseModels.
- **Fix applied**: both handlers now wrap `EnsureWithinPoolAsync` guard-check + `SaveChangesAsync` write in one `CreateExecutionStrategy` → `BeginTransactionAsync`/`CommitAsync` transaction, taking `pg_advisory_xact_lock(lockKey)` at the top of the transaction. `lockKey = ((long)0x46554E44L << 24) ^ poolScope` where `poolScope = FundingSourceId (if >0) else ProgramId` — lock granularity mirrors the guard's two `SumAsync` branches (per-source vs program pool). Auto-releases at transaction end; **no schema change**. `catch (BadRequestException) { throw; }` added before generic catches to preserve the guard's user-facing "exceeds available funds" message.
- **Verify**: two concurrent disbursements against a pool with room for one → exactly one succeeds. (`Base.Application` builds clean, 0 errors.)

### [x] C-2 — Uncapped disbursement when case has no funding source AND no program — `FIXED (session 1)`
- **Where**: `ServiceLogFundingGuard.cs:71-75`, reachable via `CreateBeneficiaryServiceLog.cs:112` + `UpdateBeneficiaryServiceLog.cs:113`. (documented ISSUE-7, confirmed live)
- **Problem**: null-program early-return skipped the ceiling entirely → unlimited spend for a monetary log with no funding source AND no program.
- **Business rule chosen**: **Block — require a pool.** Money-out must trace to transferred money-in; no source + no program = no pool to charge against = unaccountable spend.
- **Fix applied**: replaced the silent `if (programId is null || programId.Value <= 0) return;` at the no-source fallback with `throw new BadRequestException("A funding source or program is required to log a monetary disbursement. …")`. Non-monetary logs (`amountCents <= 0`) still early-return at line 28; source-attributed logs still cap at lines 32-63; program-attributed logs fall through to the existing program-pool branch (69-107). No schema change. Both guard callers (Create/Update) already pass `ProgramId` + `FundingSourceId` + `AmountCents`, and both re-throw `BadRequestException` so the message surfaces to the user.
- **Verify**: a monetary service log with neither funding source nor program is now rejected with a clear message instead of silently escaping the cap. (`Base.Application` builds clean, 0 errors.)

## HIGH

### [x] H-1 — Verification bypass ①: enrolment auto-promotes DFT→ACT skipping VRF — `FIXED (session 2)`
- **Where**: `UpdateBeneficiary.cs:167-188`, `CreateBeneficiary.cs:261-283`; enrolment modal → UPDATE_BENEFICIARY_MUTATION; `CreateCase.cs:79` accepts ACT.
- **Fix applied**: removed both enrolment-driven auto-promote blocks (Create + Update). ### [x] H-2 — Verification bypass ②: client-postable `BeneficiaryStatusId` at Create — `FIXED (session 2)`
- **Where**: `BeneficiarySchemas.cs:289`, `CreateBeneficiary.cs:119` (Adapt), `:124` forces DFT only when `<=0`.
- **Fix applied**: Create now forces the DFT status unconditionally (loads BENEFICIARYSTATUS/DFT MasterData, throws `BadRequestException` if unseeded) — any client-supplied `BeneficiaryStatusId` is ignored, so no new beneficiary can start past Draft.

### [x] H-3 — Verification bypass ③: weak evidence check — `FIXED (session 2)`
- **Where**: `SaveBeneficiaryVerification.cs:34-41` (any single doc); `VerifiedDocumentId` ownership unvalidated; `verification-tab.tsx:194-260` fallback latest doc; `GetBeneficiaryVerification.cs:117-119` `CanMarkVerified` trivially met.
- **Fix (H-1/2/3 together)**: single server-enforced gate — status is server-derived only; verification requires validated, owned evidence; no client path promotes past VRF.
- **Fix applied**: SaveBeneficiaryVerification handler now validates every supplied `VerifiedDocumentId` (non-null, > 0) against the beneficiary's own non-deleted `BeneficiaryDocuments` before the upsert; any borrowed/stale/deleted doc id is rejected with `BadRequestException`. Check runs before the try block so the message surfaces (not swallowed by the generic catch). `dbContext.BeneficiaryDocuments` reachable because `IApplicationDbContext : ICaseDbContext`; no new injection, no schema change. Handler still does NOT change status.

### [x] H-4 — Unconstrained status transitions — `FIXED (session 3)`
- **Where**: `UpdateCaseStatus.cs:60` (any→any incl CLOSED); `update-status-modal.tsx:26-38`; `UpdateBeneficiaryStatus.cs:51-55` (no TypeCode filter/guard).
- **Fix applied**: both generic status setters now (1) resolve the target constrained to their own `MasterDataType` (`CASESTATUS` / `BENEFICIARYSTATUS`) so a cross-type MasterDataId/code can no longer land on `StatusId`/`BeneficiaryStatusId`, and (2) enforce a legal-transition guard before the write. Case: rejects same-status no-op, into-CLOSED (routed to Close Case), out-of-CLOSED (routed to Reopen Case). Beneficiary: rejects same-status no-op, any move out of a terminal state, into DFT, into VRF (owned by Verify Beneficiary), and DFT/VRF→ACT (VRF→ACT owned by CreateCase; DFT→ACT is the H-1 verification-skip). SUSPENDED stays reversible (excluded from TerminalStates) so suspend/reactivate lateral moves remain legal. No schema change.

### [x] H-5 — Delete/reactivate leaks committed spend — `FIXED`
- **Where**: `DeleteCase.cs:30-31` (no cascade); `ToggleBeneficiaryServiceLog.cs:26` (reactivation bypasses guard).
- **Fix applied**: (1) `DeleteCase` now cascades the soft-delete to the case's committed spend — after marking the case deleted it loads the case's active `BeneficiaryServiceLogs` (`CaseId == caseId && !IsDeleted && IsActive`) and sets `IsActive=false; IsDeleted=true` on each, so those funds are returned to the program/source pool instead of being orphaned as active-forever commitments. (2) `ToggleBeneficiaryServiceLog` now re-runs `ServiceLogFundingGuard.EnsureWithinPoolAsync` on the reactivation (false→true) transition only, excluding the log's own id, and lets `BadRequestException` propagate — so re-committing a previously-freed log cannot push the pool over cap. Deactivation still frees funds unchecked. No schema change.

### [x] H-6 — Payment-mode required fields FE-only — `FIXED`
- **Where**: `service-log-tab.tsx:393-406` (FE enforced); BE lenient. Add BE validation.
- **Fix**: New `ServiceLogPaymentGuard.EnsurePaymentDetailsAsync` re-applies the FE's by-mode required-field rules server-side, wired into both Create and Update handlers before the funding guard. Fires ONLY when a `PaymentModeId` is supplied (legacy/mode-null rows untouched); Cash exempt; family detected via the same uppercased `code + " " + name` substring match the FE uses — Bank requires BankName+BankAccountNumber, UPI requires UpiId, any other traceable mode requires PaymentReference. Throws `BadRequestException`. Build clean (0 errors). Conditional-guard reconciliation preserves the intentional Create/Update validator leniency per delegated authority.

### [x] H-7 — Inconsistent enrolled count / capacity % — `FIXED`
- **Where**: `GetAllPrograms.cs:140-144,163-165` (ENROLLED+GRADUATED+EXITED+SUSPENDED) vs `GetProgramById.cs:94,99-101` (ENROLLED only). Pick one definition, apply to both.
- **Fix**: Adopted the grid's canonical slot-occupier definition (`slotStatuses = ENROLLED + GRADUATED + EXITED + SUSPENDED`, WAITLISTED excluded/own count) in `GetProgramById.cs` so `EnrolledCount` and the derived `CapacityPercent` on the detail view now match the grid card. Previously the detail counted `ENROLLED` only, so COMPLETED programs (everyone GRADUATED/EXITED) read 0 beneficiaries / 0% while the card showed the true figure. Mirrored the rationale comment. Build clean (0 errors).
