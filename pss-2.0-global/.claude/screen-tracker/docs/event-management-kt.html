▸ Overview

What is it? Event Management is how the NGO runs real-world and virtual events — a charity gala, a walkathon, a webinar. An organiser creates the event (what, when, where), adds the tickets people can buy, designs the e-ticket attendees receive, publishes a public registration page, and then watches registrations and payments come in — right up to checking guests in at the door and closing the event. It is not one screen but a chain that work together, with payment gateways handling the money for paid tickets. Many fundraising events also run an auction (#48) — items donors bid on during the gala — which lives inside the event as its own sub-screen.

This guide has only four parts: ① the flow, ② a live animated example, ③ the test cases, and ④ the data tables. Nothing else to memorise.

1 The Flow — Start to End

An event has its own 5-status lifecycle, but reaching "live" means filling in all four tabs of the event screen first.

The event's five statuses

StatusMeaningWhat moves it on
DraftBeing built. Not visible to the public, can't take registrations.Fill required fields → Save & Publish
PublishedLive and visible. The registration page is open and tickets can sell.The start date arrives
In ProgressThe event is happening right now. Check-in is active.Derived — never a stored value
CompletedFinished. Read-only; feeds post-event analytics.Terminal (after Complete Event)
CancelledCalled off. No registrations processed.Terminal (needs a Cancellation Reason)
⚠ Two things testers always trip on. ① "In Progress" is calculated, never saved — the event shows In Progress automatically when now is between Start and End dates and it is Published. You won't find it as a stored status. ② Cancelled & Completed are dead ends — once there, an event can't go back. Cancelling requires a Cancellation Reason.

The journey in plain words (one screen, four tabs)

What used to be three screens — Event #40, Ticketing #46 and Registration Page #169 — is now one screen with four tabs. You work left → right; each tab saves on its own, and tabs 3 & 4 unlock only after the event exists (i.e. once Tab 1 is saved). The e-ticket template (#176) and the auction (#48) stay as their own screens.
  1. Tab 1 — Event basics. Draft Name, mode (In-person / Virtual / Hybrid), category, dates & host org-unit. Save creates the event as Draft with an auto code (GALA-2026) — and unlocks the remaining tabs. (Existing structure — unchanged.)
  2. Tab 2 — Venue & details. Where it happens (venue address or virtual meeting link), schedule and description. (Existing structure — unchanged.)
  3. Tab 3 — Ticket configuration. Define ticket types & prices — "VIP Table" $5,000, "Standard Seat" $250 — each with quantity and a sale window. Each ticket can point at an e-ticket template (designed separately in #176). (Was screen #46.)
  4. Tab 4 — Registration page setup. → Published The public slug (/event/charity-gala-2026), branding, payment gateway & methods — plus the event content (speakers, custom questions, suggested amounts) that used to sit on the old tab 3. (Was screen #169 + the old content tab.)
  5. Publish. Published Publishing flips the event to Published and the page to Active; the public URL goes live.
  6. Attendees register & pay. A visitor on the public page picks tickets, fills the form and pays through the gateway. Each purchase creates an EventRegistration and sends the e-ticket.
  7. Run the event. In Progress On the day, staff use QR check-in to mark attendees as arrived. An auction (#48) may run alongside — see its own section.
  8. Close it. Completed Complete Event locks everything read-only and opens post-event analytics.
In plain words: a paid registration sits at Pending Payment until the gateway confirms, then becomes Confirmed and the e-ticket sends. If the ticket type is sold out, the visitor goes to the Waitlist and is promoted when someone cancels. Meanwhile the page auto-flips from Published → Active when its open date arrives, and to Closed when capacity is hit or the event ends.

The auction (#48) is a separate screen that runs alongside an event — its items, bidding and winner flow are covered in the test cases and entity tables, not in this build flow.
