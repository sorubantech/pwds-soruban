Event Management — Knowledge Transfer

How an event goes from an empty draft to a published, ticket-selling, check-in-ready event — across four screens that snap together — explained simply so a new team member can start testing on day one.

Module: CRM · Events Event #40 · Ticketing #46 Ticket Template #176 Reg Page #169 + Payment Gateways

What is it? Event Management is how the NGO runs real-world and virtual events — a charity gala, a walkathon, a webinar. An organiser creates the event (what, when, where), adds the tickets people can buy, designs the e-ticket attendees receive, publishes a public registration page, and then watches registrations and payments come in — right up to checking guests in at the door and closing the event. It is not one screen but four that work as a chain, with payment gateways handling the money for paid tickets.

This guide has only four parts: ① the flow, ② a live animated example, ③ the test cases, and ④ the data tables. Nothing else to memorise.

1 The Flow — Start to End

An event has its own 5-status lifecycle, but reaching "live" means building the chain across all four screens first.

The event's five statuses

StatusMeaningWhat moves it on
DraftBeing built. Not visible to the public, can't take registrations.Fill required fields → Save & Publish
PublishedLive and visible. The registration page is open and tickets can sell.The start date arrives
In ProgressThe event is happening right now. Check-in is active.Derived — never a stored value
CompletedFinished. Read-only; feeds post-event analytics.Terminal (after Complete Event)
CancelledCalled off. No registrations processed.Terminal (needs a Cancellation Reason)
⚠ Two things testers always trip on. ① "In Progress" is calculated, never saved — the event shows In Progress automatically when now is between Start and End dates and it is Published. You won't find it as a stored status. ② Cancelled & Completed are dead ends — once there, an event can't go back. Cancelling requires a Cancellation Reason.

The journey in plain words (across the four screens)

  1. Create the event. #40 Organiser clicks + New Event, fills name, mode (In-person / Virtual / Hybrid), category, org unit, dates & venue, then Saves as Draft. System auto-stamps a code (GALA-2026).
  2. Add the tickets. #46 On the Ticketing screen the organiser defines ticket types — e.g. "VIP Table" $5,000, "Standard Seat" $250 — each with a price, quantity and sale window.
  3. Design the e-ticket. #176 A reusable Ticket Template (built in the rich-text editor) decides what the e-ticket / receipt looks like, using merge fields like {{RegistrantName}} and {{EventName}}. Each ticket type can point at one.
  4. Configure & publish the page. #169 The Registration Page gets a public slug (/event/charity-gala-2026), branding, and a payment gateway for paid tickets. Publishing flips the event to Published and the page to Active.
  5. Attendees register & pay. A visitor lands on the public page, picks tickets, fills the form, and pays through the gateway. Each purchase creates an EventRegistration and sends the e-ticket.
  6. Run the event. In Progress On the day, staff use QR check-in to mark attendees as arrived.
  7. Close it. #40 Complete Event locks everything read-only and opens post-event analytics.

Two smaller cycles a tester must know

A registration's life (#46 / #169)

What happens to one attendee after they hit "Register".

stateDiagram-v2
    direction LR
    [*] --> Pending: free + needs approval
    [*] --> PendingPayment: paid ticket
    PendingPayment --> Confirmed: gateway success
    Pending --> Confirmed: admin approves
    [*] --> Waitlist: over capacity
    Waitlist --> Confirmed: a seat frees up
    Confirmed --> Cancelled: cancel (frees seat)
    Confirmed --> [*]: checked in

The registration page's life (#169)

The public page has its own status, separate from the event.

stateDiagram-v2
    direction LR
    [*] --> PageDraft: configure
    PageDraft --> PagePublished: Publish
    PagePublished --> PageActive: open date reached
    PageActive --> PageClosed: full / end date / event over
    PageClosed --> PageArchived: Archive
    PageArchived --> [*]
In plain words: a paid registration sits at Pending Payment until the gateway confirms, then becomes Confirmed and the e-ticket sends. If the ticket type is sold out, the visitor goes to the Waitlist and is promoted when someone cancels. Meanwhile the page auto-flips from Published → Active when its open date arrives, and to Closed when capacity is hit or the event ends.
