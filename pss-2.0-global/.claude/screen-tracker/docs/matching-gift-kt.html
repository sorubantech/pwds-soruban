Matching Gift

Many employers double or triple their staff's charitable gifts — Microsoft 1:1 up to $15k/yr, Apple 3:1, ExxonMobil 3:1. When the NGO surfaces that, a donor's gift can be matched at zero extra cost to the donor. This screen is the operations hub: curate corporate programmes, track every match from Eligible to Received, give donors a self-service employer search, and configure auto-detection & reminders.

▸ Overview

A matching gift is an employer's promise to match an employee's charitable donation. The NGO's job is to surface that money — let donors discover their eligibility, and chase each match through to the bank. It's "free" revenue the donor already unlocked simply by giving.

One screen, three entities

MatchingCompany is the curated catalogue of corporate programmes (Tab 1). MatchingGift is one tracked opportunity tying a donation to a company (Tab 2). MatchingGiftSettings is the single per-tenant config row (Tabs 3 & 4).

All three live under schema fund, under one menu with four tabs. The screen consumes existing entities — GlobalDonation, Contact, EmailTemplate — as FK sources; it doesn't recreate them.

The match math

When a match is created, the amount is computed once and frozen:

  • MatchAmount = DonationAmount × MatchRatio — snapshotted at create
  • If the company later changes its ratio, existing matches keep their original figure
  • MinimumDonation gates eligibility; MaximumPerYearPerEmployee caps the yearly total
  • Ratio is stored decimal(4,2) (1.0, 2.0, 3.0) and shown "1:1", "2:1", "3:1"
Why it matters. A $500 gift to a Microsoft (1:1) employee becomes $1,000 for the cause; to an ExxonMobil (3:1) employee, $2,000 of match on top. The donor pays nothing extra — the NGO just has to know the programme exists, tell the donor, and file the paperwork. This screen does all three.

1 The Flow — Start to End

The whole matching-gift journey, in order. The admin curates the programme, a donation lands, a match record is born, the donor files with the employer, the employer approves, the funds arrive — and the original gift is doubled.

  1. Curate the programme. In Tab 1, add the donor's employer — Microsoft, ratio 1:1, cap $15,000/yr, eligibility (full-time), submission URL & steps. This is reference data, curated once and reused for every employee.
  2. A donation lands. A donor gives (e.g. $500 via the donation flow). If Auto-detect employer is on and their contact record names a matching company, the screen flags the match opportunity; otherwise an admin links it manually.
  3. Match created. Eligible A MatchingGift row is born tying the GlobalDonation to the MatchingCompany. The donation amount is snapshotted and MatchAmount = $500 × 1.0 = $500 is computed and stored.
  4. Donor submits to employer. Submitted The donor files the match request on the employer's giving portal (the NGO can send a reminder email — a service placeholder). Admin marks it Submitted with the date; ExpectedDate is set.
  5. Employer approves. Approved The employer green-lights the match. Admin marks Approved; ApprovedDate stamped. (Or the employer declines → Rejected with a reason ≥ 10 chars.)
  6. Funds received. Received The matched cheque arrives. Admin marks Received; ReceivedDate stamped and the MatchAmount now counts in the "Matches Received" KPI. Terminal state.
  7. Off-path outcomes. A match left too long, or past the yearly cap, can go Expired. A Rejected match can be Resubmitted — that clones a fresh Eligible row to try again.
In plain words: the company catalogue is built once; from then on every eligible donation spawns a match record that the team walks through four states. The only money that actually moves is the employer's cheque at "Received" — everything before it is tracking and nudging so that cheque doesn't get forgotten.

5 Statuses & Lookups

The match lifecycle is six MasterData rows (typeCode MATCHINGGIFTSTATUS, codes UPPERCASE). UpdateMatchingGiftStatus is a state machine — it auto-stamps the matching date column and rejects illegal jumps.

Match lifecycle

StatusMeaning
EligibleMatch identified; not yet filed with the employer (amber-highlighted in the grid)
SubmittedDonor filed the request; ExpectedDate set
ApprovedEmployer green-lit the match
ReceivedCheque in the bank — counts in KPI (terminal)
RejectedEmployer declined; reason ≥ 10 chars (resubmittable)
ExpiredWindow passed or yearly cap exceeded (terminal)

Guarded transitions

From→ Allowed
EligibleSubmitted · Expired
SubmittedApproved · Rejected · Expired
ApprovedReceived · Rejected
Received— (terminal)
RejectedResubmit → new Eligible row (clone)
Expired— (terminal)

Anything else → 400 "Cannot transition from {current} to {new}".

The match math, in detail

MechanicHow it works
MatchAmountDonationAmount × MatchRatio, computed at create and frozen — later ratio changes don't recompute existing matches
MatchRatio storagedecimal(4,2): 1.0 → "1:1", 2.0 → "2:1", 3.0 → "3:1", 0.5 → "0.5:1"
MinimumDonationGift must be ≥ this to qualify (drives auto-detect eligibility)
MaximumPerYearPerEmployeeYearly cap; approaching it can auto-Expire the match. Validation: max ≥ min when both set
MinimumDonationForReminderSettings threshold (default $25) — only gifts above it trigger reminder emails

Status is FK MasterData, not a string enum. Unlike Crowdfunding's string PageStatus, the match status is a real MasterData FK (MatchingGiftStatusId), seeded UPPERCASE. Code that compares against PascalCase ("Eligible") silently misses — match on the code (ELIGIBLE) or the resolved name, the same trap noted across the case-management screens.
