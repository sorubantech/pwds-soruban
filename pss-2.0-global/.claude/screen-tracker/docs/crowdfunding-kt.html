Crowdfunding

The NGO runs a goal-and-deadline campaign for a specific need — build a school, fund a clinic, respond to an emergency. One public page tells the story, shows a thermometer, lists impact tiers and milestones, and takes donations directly. Admins manage the portfolio from a CRM grid; the rich page is built in a 6-tab editor and served to anonymous donors at /crowdfund/{slug}.

▸ Overview

Crowdfunding is org-led fundraising: the NGO defines a concrete need, sets a goal and deadline, and collects donations on one rich public page. (Contrast with P2P, where supporters raise on the NGO's behalf.)

The two surfaces, one entity

#16 is the back-office: a cards-grid that creates the campaign (8-field Quick-Create), shows KPIs, and drives the lifecycle. It introduces the fund.CrowdFunds entity end-to-end but keeps its UI minimal.

#173 is the rich layer on top — a 6-tab editor for the full story/design, plus the donor-facing public page. It wraps the same entity: no new table, just more handlers and two public endpoints.

Where the money flows

Every gift is one fund.GlobalDonation row marked DonationType = 'Crowd Donation'. Attribution is via a junction:

  • The donate flow inserts the GlobalDonation and a fund.CrowdFundDonations row in one transaction
  • That junction row links CrowdFundId ↔ GlobalDonationId (one-to-one on the donation)
  • Totals are aggregates over the junction — skip the junction insert and every total reads 0
Two screens, one feature. #16 built the engine — the CrowdFunds entity, the summary, the Quick-Create and lifecycle. #173 is the experience on top: the 6-tab content editor and the public donor page. Nothing in #173 creates new entities — it reads and writes what #16 already ships. The "child" data (milestones, updates, impact tiers, FAQ) lives as jsonb arrays on the entity itself, not in separate tables.

1 The Flow — Start to End

The whole crowdfunding journey, in order. The admin creates and configures the campaign, publishes it, donors give on the public page, milestones tick over, the goal is met, and the campaign closes.

  1. Create the campaign. Draft In #16's Quick-Create, fill 8 fields — name, slug, goal, currency, start/end dates, donation purpose, category. Saving creates the fund.CrowdFunds row as Draft and opens its drawer.
  2. Configure the page. Draft In #173's 6-tab editor, write the story, add impact tiers & FAQ, set amount chips & payment methods, define milestones, draft updates, and pick theme colours / section toggles — all against the same row, with a live preview.
  3. Publish. Published / Active Validation gates the button (name, slug, goal > 0, dates, purpose, ≥1 payment method, ≥1 amount chip or custom-allowed). Publishing stamps PublishedAt and — if the start date has passed — auto-advances to Active. /crowdfund/{slug} goes live.
  4. Donors give. A visitor lands on the public page, picks an amount (chip or custom), optionally makes it monthly / covers fees / stays anonymous, and donates. Each gift becomes a GlobalDonation + a CrowdFundDonations junction row.
  5. Milestones tick over. As the total climbs, any milestone whose amount is crossed auto-flips to Reached (unless the admin pinned its status). The thermometer, donor wall and updates feed all refresh from the aggregates.
  6. Goal met. Goal Met When Total Raised ≥ Goal, the page shows a "🎉 Goal met!" banner. What happens next is the tri-state: Keep Accepting (donations continue), Auto-Close (flips to GoalMet, donate disabled), or Show Stretch Goal (a second thermometer to the stretch amount).
  7. Close. Closed At the end date or via "Close", the campaign goes Closed — the public page turns read-only with an "ended" banner showing the final total. Later it can be Archived (410 Gone) or Duplicated into a fresh Draft.
In plain words: the admin builds the story once, presses Publish, and the public page does the rest — every donation is tagged to the campaign the moment it lands, so the thermometer, milestones and donor wall are always just sums. The only ongoing job is posting updates and deciding what to do when the goal is beaten.

3 The Surfaces in Detail

What each screen looks like and does. One CRM cards-grid (#16), one 6-tab editor and one public donor page (#173).

📋 #16 — Crowdfunding management grid

KPIs & grid

  • 4 KPIs — Active Campaigns, Total Raised, Total Donors, Goals Met (+ completion-rate %)
  • Search + 4 chips — All / Active / Goal Met / Draft (with counts)
  • Cards grid — each card: category-gradient hero, name, progress bar, raised/goal, donors, end date
  • 560px drawer — summary, 4 performance tiles, milestones, recent donors, latest updates, quick actions

Create & row actions (status-conditional)

  • Quick-Create — 8 fields → a Draft; Quick-Edit the basics (slug locks once live)
  • Active → Edit (→ #173) · View Page · Dashboard
  • Draft → Edit · Preview · Publish · Delete (guards: Draft-only + no donations)
  • Goal Met / Closed / Archived → View · Close / Duplicate / Archive
The card hero is colour-coded by category — Education (teal), Healthcare (violet), Emergency (red/amber), Environment (sky/green), Community (orange), Other (slate) — each with its own Lucide icon. The progress bar shifts colour by completeness (red < 50% · amber 50–75% · green ≥ 75% · 🔥 green when ≥ 100%).

🎨 #173 — The 6-tab page editor

The editor lives at setting/publicpages/crowdfundingpage. A card-list view (reusing #16's KPIs/chips/cards) precedes it; opening a card switches to the tabbed editor with a live-preview pane (desktop/mobile, 300ms debounce). Each tab saves its own fields (partial update via updateCrowdFundPage); the footer carries Save Draft · Preview · 🚀 Publish.

🌐 #173 — The public donor page · /crowdfund/{slug}

Left column — the story

  • Org navbar + gradient hero (category icon, or hero image/video)
  • Headline + org/category tags + rich-text story
  • Your Impact tiers, Milestones timeline (Reached/InProgress/Upcoming)
  • Updates feed, Recent Supporters donor wall, FAQ accordion — each gated by its section toggle

Right column — sticky donate widget

  • Progress widget — "$38,400 raised of $50,000", thermometer, "345 donors · 52 days left"
  • Donate form — amount chips + Custom, make-it-monthly, cover-fees (+2.9%+$0.30), anonymous, name/email
  • Donate Now → initiateCrowdFundDonation → gateway → thank-you state
  • Share — Facebook · X · WhatsApp · LinkedIn · Copy Link
Status gating & SSR. The route is anonymous and server-rendered with generateMetadata for OG tags and ISR revalidate: 60. Draft → 404 (unless a preview-token is present, which shows a "NOT YET LIVE" banner); Active/Goal Met/Closed render (Closed disables the form); Archived → 410 Gone. Donations are rate-limited (5/min/IP/slug), CSRF-protected and honeypot-guarded.
Service placeholders. Real Stripe/PayPal hand-off is stubbed — initiateCrowdFundDonation returns a mocked payment session until a gateway is connected in companypaymentgateway. Receipt email, WhatsApp donation alerts, the per-card "Dashboard" action and reCAPTCHA scoring are also stubbed (toast / log) until those services land.

4 Entity Tables — Data Design

The whole module is one new entity — fund.CrowdFunds — plus a thin junction. There are no child tables: milestones, updates, impact tiers, FAQ, amount chips and section toggles all live as jsonb on the entity. Donations attach through fund.CrowdFundDonations.

5 Statuses & Lookups

The crowdfunding lifecycle is a string enum on the entity (not MasterData). The same six states drive both screens; the public page renders or 404/410s based on them.

Campaign lifecycle

StatusMeaning
DraftBeing built; public route 404s (preview-token shows a banner)
PublishedPublish passed validation; live once the start date arrives
ActiveAccepting donations (auto from Published when StartDate ≤ now)
Goal MetTotal ≥ Goal — reached automatically (Auto-Close) or shown as a badge
ClosedEnd date or "Close" — page read-only with final total
ArchivedHidden from the list; public route returns 410 Gone

Lifecycle transitions (guarded)

ActionFrom → To · guard
PublishDraft → Published (auto → Active if started); validation gate
UnpublishPublished → Draft · only if zero donations
CloseActive / GoalMet → Closed
ArchiveClosed / Draft / GoalMet → Archived
Duplicateany → new Draft ({slug}-copy, dates & donations cleared)
DeleteDraft only · and no donations exist (soft-delete)

The goal-exceeded tri-state

GoalExceededBehaviorWhat happens when Total ≥ Goal
KeepAccepting (default)Donations keep flowing; a "🎉 Goal met!" badge shows but status stays Active
AutoCloseFlips PageStatus to GoalMet and disables the donate form
ShowStretchGoalA second thermometer appears toward StretchGoalAmount; donations continue

Other enumerations

LookupValues
CampaignCategoryEducation · Healthcare · Emergency · Environment · Community · Other
CurrencyUSD · EUR · GBP · INR (+ tenant default)
Milestone statusReached · InProgress · Upcoming
Payment methods (EnabledPaymentMethodsJson)stripe · paypal (Apple/Google Pay · bank transfer = future)
Font familySystem · Georgia · Poppins · OpenSans

Badge vs stored status. The "Goal Met" badge is computed live from TotalRaised >= GoalAmount, so a campaign can show the badge while PageStatus is still Active (only Auto-Close, an admin Close, or a job actually mutates the status). The KPI / chip "Goal Met" count therefore includes PageStatus = 'GoalMet' or (Active/Closed with Total ≥ Goal).
