Case Management — Knowledge Transfer

The full journey from setting up a Program, enrolling a Beneficiary, to opening, working and closing their Case — explained with one running example so a new team member can start testing on day one.

Module: CRM · Case Management Screen #50 · FLOW Program → Beneficiary → Case 5-status lifecycle

What is it? Case Management is where social workers intake a beneficiary's need (a child needing schooling, a medical issue, housing aid…), track the work done on it (action plans, notes, referrals, documents), and finally close it with an outcome. But a case never stands alone — it always sits at the end of a chain: a Program (e.g. an education sponsorship) defines what is offered → a Beneficiary is enrolled into it → a Case is opened to actually deliver and track that help.

Two layers, one story. The program / beneficiary layer records what was actually delivered — Service Logs (tuition paid, food given) and Milestones (passed a grade). The case layer records the casework around it — the Action Plan (planned steps), History (the diary), Documents (evidence) and Referrals (needs sent outside the program). The link is causal & manual: the worker plans an action → delivers it → records the matching Service Log / Milestone. Every delivery should trace back to a planned action and be backed by a document.

This guide has five parts: ① the Program → Case flow, ② a live example with three lifecycle variants, ③ the test cases, ④ the data tables, and ⑤ ready-to-run seed data.

1 The Flow — Program to Case

First the full chain (how a case comes to exist), then the five-status lifecycle the case itself walks.

From Program to Case — the full chain

This is what a worker produces by walking the screens once, in order. Our running example is the Hope Education Sponsorship program (₹60,000/yr tuition + ₹30,000/yr hostel + ₹8,000/mo food) and a sponsored child, Aisha Khan.

  1. Create the Program (#51) — name, category, funding model, capacity, then its Eligibility Criteria, Services Included (tuition / hostel / food), Outcome Metrics (attendance, pass rate) and assigned staff + lead. This is the menu of what the program offers.
  2. Register the Beneficiary (#49) — Aisha's intake: demographics, need category, consent. Saved with status Draft.
  3. Enrol the Beneficiary into the Program — creates a BeneficiaryProgramEnrollment with status Enrolled. This auto-promotes Aisha from Draft → Active.
  4. Open a Case (#50) — to actually deliver and track the sponsorship. Links Aisha + the program + an owner; auto-codes CASE-NNNN, status Open.
  5. Build the Action Plan — the planned steps: verify consent → pay tuition → arrange hostel → set up monthly food → collect report card → confirm next-grade promotion. Each item has an owner, a target date and a status.
  6. Deliver & record Services — as each action is done, log the matching Service Log on the beneficiary: Tuition ₹60,000, Hostel ₹30,000, Food ₹8,000 × 12 (all linked back to this case via CaseId).
  7. Track Milestones — outcomes achieved: Passed Grade 2 (Achieved), Promoted to Grade 3 (In Progress).
  8. Keep History & Documents — every visit/call/status-change is a Note (the diary); receipts, report cards and consent forms are Documents (the evidence).
  9. Refer out if needed — for a need the program can't meet (e.g. a post-graduation job pathway), raise a Referral to an outside organisation.
  10. Close at the outcome — when support concludes, Close Case with an outcome (e.g. Graduated) + closure summary; the beneficiary's enrollment moves to Graduated/Exited.
Where each Case tab gets its data. Action Plan = the planned steps (your roadmap). History = notes + status changes + dated events (the diary). Documents = uploaded evidence. Referrals = hand-offs outside the program. The beneficiary's Services & Milestones (on screen #49) are what those actions actually produced — they carry the case's CaseId so they tie back here.

The five statuses

StatusMeaningWho moves it
OpenJust created. Sitting in the queue, work not started.Auto-set when the case is created
In ProgressA case worker is actively working it — visiting, calling, delivering services.Staff → "Update Status"
PendingWaiting on someone else — usually an external referral response.Staff → "Update Status"
ResolvedThe need is met. Ready to wrap up.Staff → "Update Status"
ClosedFinal. Outcome + closure summary recorded. Read-only.Staff → "Close Case" modal
⚠ The 6th badge — Overdue — is NOT a real status. It is calculated on the fly: any case that is still Open/In Progress/Pending and whose Follow-up Date is in the past shows up red. The stored status never changes. When you test, remember: setting a follow-up date in the past makes the row turn red — that's correct behaviour, not a bug.

The journey in plain words

  1. Intake. Staff clicks + New Case, picks the beneficiary, writes a title + description, sets priority & assigned staff. On save the system auto-stamps a code (CASE-NNNN), sets status Open, and records today as the opened date.
  2. Casework. Staff opens the case file and works it — adds Notes (home visits, phone calls), builds an Action Plan (tasks with owners + due dates), and moves the status to In Progress.
  3. Referral (if needed). If outside help is required, staff raises a Referral (auto-coded REF-NNNN) to an external organisation — consent must be ticked before it can be sent — and parks the case as Pending until they respond.
  4. Resolution. Once the response is in and the need is met, status goes back to In Progress and then Resolved.
  5. Closure. Staff clicks Close Case, picks an Outcome and writes a required Closure Summary (optionally a post-close follow-up + lessons learned). The case becomes Closed and read-only.
Communication happens along the way. A welcome message to the guardian at intake, the referral hand-off to the partner org, the response coming back, and a closure note at the end — these touchpoints are shown live in the next section.

4 Entity Tables — Data Design

All in the case database schema. The case layer = one parent (Case) + four children. The beneficiary layer = Service Logs and Milestones — beneficiary-owned, but each carries an optional CaseId so it ties back to the case that delivered it. The "History" tab is built on the fly — it has no table.

⚠ Milestone status casing. MILESTONESTATUS DataValues are UPPERCASE (ACHIEVED, INPROGRESS) in the DB. Code that compares them must be case-insensitive (e.g. statusCode.toUpperCase() === "ACHIEVED") — a PascalCase === "Achieved" check silently fails and milestones drop off the timeline / turn grey.
History tab = no table. It is assembled at read-time from the case's own dates, its notes, status changes, and action-item updates — merged into one timeline. Nothing to seed or test as storage; just verify the events show in order.

5 Seed Data — Load the Example

Three ready-to-run PostgreSQL scripts under PSS_2.0_Backend/DatabaseScripts/Seed/ recreate exactly the example in this guide. All are idempotent (safe to re-run) and scoped to company 1.

FileWhat it createsCodes
seed_case_lifecycle_flows.sql ONE program with all three lifecycle variants (Rohan started / Aisha in progress / Fatima completed). Self-cleans its own rows and the old DEMO_FLOW sample first. DEMO_LIFECYCLE, DEMO_LC_*
seed_case_manual_flow.sql The same in-progress set (Aisha), written as 13 plain step-by-step INSERTs — the way a worker creates it by walking the screens. No loops / variables; each step NOT EXISTS-guarded. MAN_EDU, MAN_AISHA, MAN_CASE_AISHA
cleanup_case_management_by_program.sql Wipes the entire case-management footprint of one program (set v_program_id or v_program_code) — logs, milestones, case children, cases, enrollments, beneficiaries, program children — FK-safe. parameterised
Prereqs (master data — not seeded by these files): run Program-sqlscripts.sql, Beneficiary-sqlscripts.sql and Case_Seed.sql first, and have ≥1 app.Branches + app.Staffs row for company 1.
Cleanup keeps shared beneficiaries safe. A beneficiary enrolled in more than one program is not deleted by default — only their rows for the target program are removed. Set v_force_all = true to delete every touched beneficiary regardless (also wipes their data in other programs).
Money: Service-Log AmountCents is stored in paise — ₹60,000 → 6000000, ₹8,000 → 800000.
