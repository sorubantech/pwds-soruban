# PSS 2.0 — Screen Testing & Auto-Fix Working Flow

> **Audience:** Management / engineering leads
> **Purpose:** End-to-end explanation of how Claude Code, Playwright, and the AI agent pipeline work together to test 121+ generated screens and auto-fix failures.
> **Scope:** From `claude reads a screen spec` → `runs Playwright tests` → `analyzes failures` → `dispatches fix agents` → `re-tests until green`.

---

## 1. The Problem We're Solving

We have **~121 screens** generated by an AI pipeline. Most are `COMPLETED`; ~10–12 are still mid-build. Manual QA on this surface area is infeasible.

We need:

1. Automated browser tests per screen (CRUD, validation, grid, widgets).
2. A way to **author tests from the same spec** the AI used to build the screen — no duplicate work.
3. A way to **feed failures back to the AI agents** that built the screen, so they self-heal.
4. **Cost control** — Claude tokens are billed per input/output token, so we must avoid re-reading huge specs on every retry.

---

## 2. The Skill Family (Slash Commands)

Five user-facing commands cover the lifecycle. Each is a **Claude Skill** — a markdown procedure Claude follows when the user types the slash command.

| Stage | Command | Purpose |
|-------|---------|---------|
| **Plan** | `/plan-screens` | Convert raw business spec → structured prompt file (12 sections) |
| **Build** | `/build-screen #N` | Drive BA → Solution Resolver → UX → BE Dev → FE Dev agents to generate code |
| **Continue** | `/continue-screen #N` | Resume a built screen for human-reported bugs / tweaks |
| **Test** | `/test-screen #N` | Author + run Playwright tests for one screen |
| **Test Batch** | `/test-batch wave N` | Run tests for many screens; single summary at end |
| **Fix** | `/test-fix #N` | Read latest failures → dispatch BE/FE agents → optionally loop until green |

```
        ┌─────────────────┐      ┌──────────────────┐      ┌─────────────────┐
        │ /plan-screens   │ ───▶ │  /build-screen   │ ───▶ │  /test-screen   │
        └─────────────────┘      └──────────────────┘      └─────────┬───────┘
                                         ▲                          │
                                         │ (human-reported          │ failures?
                                         │  bugs)                   ▼
                                 ┌──────────────────┐      ┌─────────────────┐
                                 │ /continue-screen │      │   /test-fix     │
                                 └──────────────────┘      └─────────┬───────┘
                                                                    │
                                                          --until-green loops back
                                                          to /test-screen (max 3×)
```

## 3. The File Model (What Lives Where)

### Per-screen artifacts

```
.claude/screen-tracker/prompts/
  branch.md                    ← Spec (12 sections, ~600 lines). Read by ALL agents.
  branch.test-result.md        ← Test history (newest run first). Read ONLY by test/fix skills.

PSS_2.0_Backend/...            ← Generated backend code
PSS_2.0_Frontend/...           ← Generated frontend code

tests/screens/
  branch.spec.ts               ← Playwright test file authored by /test-screen
```

### Why we split spec ↔ test history

The spec file (`branch.md`) is already **~600 lines**. Every agent (BA, BE Dev, FE Dev, Solution Resolver, UX Architect) reads it. If we appended every test failure to that file, every agent read would balloon in cost.

**Decision:** Test history lives in a sibling file `{screen}.test-result.md`. The spec file gets only a **one-line pointer** per test session in its Build Log (§⑬):

```markdown
- Session 3 — 2026-05-21 — TEST — NEEDS_FIX — see [branch.test-result.md] — 3 failures
```

### Shared infrastructure

```
tests/
  playwright.config.ts         ← One config, unbounded registry of all *.spec.ts
  shared/
    auth.setup.ts              ← Login once, save storageState.json
    fixtures.ts                ← Custom Playwright fixtures
    grid-helpers.ts            ← FlowDataTable / AdvancedDataTable interactions
    form-helpers.ts            ← Field fill, validation assertions
    test-data.ts               ← Test data prefixed with _TEST_{run-id}_ for isolation

test-results/
  results.json                 ← Playwright machine-readable output (Claude reads this)
  artifacts/                   ← Screenshots, videos, traces per failed test

playwright-report/             ← HTML report for humans
```

## 4. End-to-End Lifecycle — Stage by Stage

### Stage A — Authoring Tests (`/test-screen #41`)

**Trigger:** User types `/test-screen #41` (or `/test-screen "Branch"`).

| Step | What Claude Does | Tool Fired | I/O |
|------|------------------|-----------|-----|
| A1 | Locate screen in registry | `Grep` (single row — never `Read` the ~175K-token file) | `.claude/screen-tracker/REGISTRY.md` |
| A2 | Read full spec (12 sections) | `Read` | `.claude/screen-tracker/prompts/branch.md` |
| A3 | Check if test-result file exists | `Glob` | `.claude/screen-tracker/prompts/branch.test-result.md` |
| A4 | Check if spec file already has a test file | `Glob` | `tests/screens/branch.spec.ts` |
| A5 | Identify `screen_type` (MASTER_GRID / FLOW / CONFIG / DASHBOARD / EXTERNAL_PAGE / AUTH) | (in-memory) | from §① of spec |
| A6 | Load matching template | `Read` | `.claude/skills/test-screen/templates/master-grid.template.ts` |
| A7 | Plan test cases from spec §⑥ (UI/UX), §⑦ (CRUD), §⑫ (skip rules) | `TodoWrite` | internal task list |
| A8 | Generate `branch.spec.ts` | `Write` | `tests/screens/branch.spec.ts` |
| A9 | Mark Build Log entry pending | (in-memory) | — |

Output of stage A: a complete Playwright spec file, tagged with `@screen-41 @wave-1 @master-grid`, ready to execute.

---

### Stage B — Running Tests (Playwright via Bash)

**Trigger:** Stage A completes, or user runs `/test-screen #41` on an already-tested screen.

| Step | What Claude Does | Tool Fired | I/O |
|------|------------------|-----------|-----|
| B1 | Verify dev server is up (don't auto-start — respects user-managed `BaseUrlConfig.ts`) | `Bash` | `curl -s http://localhost:3000` |
| B2 | Run Playwright filtered to this screen | `Bash` | `pnpm playwright test --grep @screen-41 --reporter=list,json` |
| B3 | **Claude is idle during the test run** | — | **0 tokens consumed while waiting** |
| B4 | When Bash returns, read structured results | `Read` | `test-results/results.json` |
| B5 | If `--bg` flag used, run background | `Bash run_in_background:true` | user can keep chatting; Claude is paged on completion |

> **Token economics callout (for management):**
> Claude bills per token *in/out*, **not** wall-clock. During the Bash subprocess wait, **no LLM turn is active = $0**.
> A 90-second Playwright run costs only the input tokens to start the Bash call + the output tokens to parse the JSON afterward — typically <2K tokens combined.

---

### Stage C — Analyzing Results

**Trigger:** Stage B Bash returns.

| Step | What Claude Does | Tool Fired | I/O |
|------|------------------|-----------|-----|
| C1 | Parse `results.json` — passed / failed / skipped counts | `Read` | `test-results/results.json` |
| C2 | For each failure: extract test name, expected vs actual, error stack, screenshot path | (in-memory) | — |
| C3 | Cross-reference with spec §⑥ / §⑦ to add **diagnosis hints** (e.g. "form schema mismatch") | `Read` | re-read relevant spec sections only |
| C4 | Categorize skipped tests against §⑫ SERVICE_PLACEHOLDER notes | (in-memory) | — |
| C5 | Write/append new Run entry to test-result file (newest first) | `Edit` or `Write` | `.claude/screen-tracker/prompts/branch.test-result.md` |
| C6 | Append **one-line** pointer to spec §⑬ Build Log | `Edit` | `.claude/screen-tracker/prompts/branch.md` |
| C7 | Update spec frontmatter: `status: COMPLETED` → `NEEDS_FIX` if any failures | `Edit` | `.claude/screen-tracker/prompts/branch.md` |
| C8 | Summarize to user (≤8 lines): X passed / Y failed / Z skipped + suggest `/test-fix #41` | (chat) | — |

```markdown

last_run_status: NEEDS_FIX
```

### Stage D — Fixing Failures (`/test-fix #41`)

**Trigger:** User runs `/test-fix #41` (or `/test-fix #41 --until-green`).

| Step | What Claude Does | Tool Fired | I/O |
|------|------------------|-----------|-----|
| D1 | Read **only the latest run entry** of the test-result file (~50 lines, not full spec) | `Read` | `branch.test-result.md` first 50 lines |
| D2 | Read only the spec sections referenced by diagnosis hints (e.g. §⑥, §⑦) | `Read` | targeted offsets in `branch.md` |
| D3 | For each failure, classify: **BE bug / FE bug / cross-cutting** | (in-memory) | — |
| D4 | Spawn agent(s) — Sonnet by default (user is cost-conscious) | `Agent subagent_type:backend-developer` and/or `Agent subagent_type:frontend-developer` | each agent gets ONLY the failure context + relevant spec section |
| D5 | Agents edit code | (delegated `Edit` / `Write` inside agent) | `PSS_2.0_Backend/...`, `PSS_2.0_Frontend/...` |
| D6 | Verify build | `Bash` | `dotnet build` and/or `pnpm build` |
| D7 | If `--until-green`: re-invoke `/test-screen #41` automatically | `Skill skill:test-screen` | loops back to Stage B |
| D8 | Append "FIX" session entry to `branch.test-result.md` + one-line pointer to spec §⑬ | `Edit` | both files |

### The `--until-green` loop (max 3 cycles)

```
            ┌──────────────────┐
            │  /test-fix #41   │
            │   --until-green  │
            └────────┬─────────┘
                     │
       ┌─────────────▼─────────────┐
       │  Cycle 1: read failures   │
       │  → spawn BE/FE agent      │
       │  → re-run /test-screen    │
       └─────────────┬─────────────┘
                     │
              ┌──────▼──────┐
              │   green?    │── yes ──▶ status: COMPLETED, exit
              └──────┬──────┘
                     │ no
       ┌─────────────▼─────────────┐
       │  Cycle 2 (same loop)      │
       └─────────────┬─────────────┘
                     │
              ┌──────▼──────┐
              │   green?    │── yes ──▶ status: COMPLETED, exit
              └──────┬──────┘
                     │ no
       ┌─────────────▼─────────────┐
       │  Cycle 3 (same loop)      │
       └─────────────┬─────────────┘
                     │
              ┌──────▼──────┐
              │   green?    │── yes ──▶ status: COMPLETED, exit
              └──────┬──────┘
                     │ no
                     ▼
       Stop. status: NEEDS_FIX. Surface to user with diff summary
       of what was tried across 3 cycles so a human can intervene.
```

Why 3? Empirically two retries catches transient races and typo-level mistakes; three cycles is the point at which the agent is most likely going in circles. Cheaper to escalate to a human than burn a 4th cycle.

---

### Stage E — Batch Runs (`/test-batch wave 1`)

For testing groups of screens (e.g. all wave-1 screens) with a single aggregate summary.

| Step | What Claude Does | Tool Fired | I/O |
|------|------------------|-----------|-----|
| E1 | Resolve the wave / filter to a list of screen IDs | `Grep` (status rows only — never `Read` the ~175K-token file) | `.claude/screen-tracker/REGISTRY.md` |
| E2 | For each screen, ensure a spec file exists, generate `.spec.ts` if missing | `Glob`, `Read`, `Write` | one batch authoring pass |
| E3 | **Single Playwright run** filtered to all tagged screens | `Bash` | `pnpm playwright test --grep "@wave-1"` |
| E4 | Parse `results.json` once | `Read` | `test-results/results.json` |
| E5 | Per screen: update its own `{screen}.test-result.md` | `Edit` × N | one file per screen — no cross-contamination |
| E6 | Per screen: update §⑬ pointer + frontmatter status | `Edit` × N | — |
| E7 | Print aggregated dashboard to user | (chat) | `15 screens | 12 PASS | 2 NEEDS_FIX | 1 BLOCKED` |

CLI variants:
- `/test-batch wave 1` — explicit wave
- `/test-batch --new` — every screen with no test-result file
- `/test-batch --failed` — every screen currently `NEEDS_FIX`
- `/test-batch all` — every COMPLETED screen

## 6. Cost & Performance Notes (for Management)

| Concern | Mitigation |
|---------|-----------|
| Re-reading huge specs on every retry | **File split** — fix agents read only the targeted `{screen}.test-result.md` Run entry + spec sections referenced in diagnosis hints |
| LLM idle during Playwright run | **No cost** — Claude turn ends when Bash starts; resumes only when subprocess returns |
| Model choice | **Sonnet only** for the testing pipeline — `/test-fix` is hardcoded to Sonnet, no Opus override flag. Hard failures escalate to a human, not to a more expensive model. |
| Per-screen vs. batch runs | `/test-batch` shares one Playwright process across N screens — one cold-start, one results.json parse, one summary |
| Repeated fix-test loops | `--until-green` capped at **3 cycles** to bound worst-case cost |
| Test data pollution | All seed data prefixed `_TEST_{run-id}_` and cleaned up in `afterAll` |
| Auth setup per test | `auth.setup.ts` runs **once per Playwright invocation**, persists to `storageState.json`; every spec reuses it |

## 7. Claude Tool Inventory (All Tools Used in This Flow)

| Tool | Purpose in this flow |
|------|---------------------|
| `Read` | Read spec files, results.json, test-result files |
| `Write` | Create new `.spec.ts` files, new `.test-result.md` files |
| `Edit` | Append run entries, update frontmatter status, append §⑬ pointers |

| `Agent subagent_type:frontend-developer` | Dispatch FE fix — Sonnet only, no Opus override |
| `Skill skill:test-screen` | `--until-green` re-invokes the test skill in-loop |
| `TodoWrite` | Track multi-step plans (author all test cases, fix N failures) |
| `AskUserQuestion` | Disambiguate when registry has multiple matches or when scope is unclear |

## 8. What's NOT Automated (Intentionally)

- **BASE_URL** — User toggles ports in `BaseUrlConfig.ts` manually. The skill reads it; never edits it.
- **Dev server start** — We don't auto-`pnpm dev`; the skill assumes the user already has it running, and `curl`-checks before the Playwright invocation.
- **Spec changes** — If a fix requires changing the screen contract (new field, new FK), `/test-fix` **refuses** and asks the user to run `/plan-screens #N` to revise the blueprint first. Same guardrail as `/continue-screen`.
- **More than 3 fix cycles** — Surfaces to human. Loops never silently extend.
