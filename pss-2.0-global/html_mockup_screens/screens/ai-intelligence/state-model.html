Refresh scenario · "did I lose my chat?"
Before refresh
User is mid-conversation. 4 messages exchanged. URL carries the session GUID.
pss.app/aida?session=8f3e1a4c-…-d2b9
Redis: warm · last 4 turns compiled
Postgres: 4 ChatMessages rows persisted
User hits F5
Browser drops everything. New page load with the same URL. Server has no idea the refresh happened.
Redis: still warm (unaware)
Postgres: unchanged
After refresh
Frontend reads ?session=8f3e1a4c-… from URL → GraphQL: chatSession(id) → returns last 50 messages → thread rendered.
User sees their full conversation. Continues typing.
Redis: still warm · reused
Postgres: read 4 rows
The only thing actually lost: if the LLM was streaming a reply at the moment of refresh, the partial response is dropped (it was never committed). User can simply resend the last question or use Regenerate. Everything that was completed survives.
Identifiers · keys you'll see in the database
Hierarchy
SessionId (UUID) — one conversation ├── MessageId (bigserial) — each user/AI turn │ ├── AuditId (bigserial) — each LLM call (≥1 on retry) │ └── ToolCallId (uuid) — inside ToolCallsJson ├── EventId (UUID, nullable) — hook trigger that opened session └── CompanyId · UserId (BIGINT) — tenancy keys (on every row)
SessionId is the durable key. Frontend stores it in URL + localStorage. Backend uses it for memory partitioning, audit grouping, and resume-from-history.
